Lore

Everything so far has vanished the moment the program ended. File handling is how a program remembers: writing results to disk, reading yesterday's data back in, processing a file someone else produced.

The mechanics are three steps — open, read or write, close — but the closing step is the one people forget, and forgetting it can mean data never actually reaches the disk. Python's answer is the with statement, which closes the file automatically no matter what happens inside, including if an error is raised. There is essentially no reason to open a file any other way.

The second half of this chapter is the more grown-up idea. Real input is dirty: files go missing, lines are malformed, a number field contains the word "absent", a divisor turns out to be zero. Code that assumes clean input crashes on the first surprise and loses everything it had already done.

Exception handling changes that. The try block contains work that might fail; the except block says what to do when it does. The program keeps running. And once you can catch errors, you can also raise your own — including custom exception classes that describe problems in the language of your program (BlankFileError, MalformedLineError) rather than in generic Python terms. That is the difference between an error message a user can act on and one that just says ValueError.

Codex

File modes

Mode Meaning
"r" read (default); errors if the file is missing
"w" write — creates or wipes the file
"a" append — add to the end, keep what's there
"r+" read and write

The destructiveness of "w" is worth respecting: opening an existing file in write mode empties it immediately, before you write a single byte.

The with pattern

with open("data.txt", "r", encoding="utf-8") as f:
    lines = [line.strip() for line in f if line.strip()]

Three things are happening: the file closes itself at the end of the block; iterating a file object yields one line at a time without loading the whole file into memory; and .strip() removes the trailing newline that every line carries. Always pass encoding="utf-8" — it makes behaviour identical across Windows, macOS and Linux.

Catching errors precisely

try:
    result = int(a) // int(b)
except ZeroDivisionError:
    print("cannot divide by zero")
except ValueError:
    print("that wasn't a number")
else:
    print(result)          # runs only if nothing failed
finally:
    cleanup()              # runs no matter what

Catch specific exceptions. A bare except: swallows everything — including typos in your own code and the user's Ctrl-C — and turns a five-second bug into a five-hour one.

Custom exceptions

class MalformedLineError(Exception):
    pass

Three lines gives you an error type that means something in your domain. raise ... from err preserves the original cause, so the traceback shows both what went wrong and what you were doing at the time.

Trial 8.5

Task 5 — Custom Exceptions for a Marks File

Two purpose-built exception classes flag domain-specific problems: BlankFileError for an empty file, and MalformedLineError for a badly shaped record. Wrapping the low-level FileNotFoundError keeps the messages consistent across all three failure modes.

CAST[5]
class BlankFileError(Exception):
    pass


class MalformedLineError(Exception):
    pass


def load_marks(path):
    try:
        with open(path, "r", encoding="utf-8") as f:
            raw_lines = [line.strip() for line in f if line.strip()]
    except FileNotFoundError as err:
        raise FileNotFoundError(f"Could not find '{path}'.") from err

    if not raw_lines:
        raise BlankFileError(f"'{path}' has no data.")

    result = {}
    for line in raw_lines:
        fields = line.split(",")
        if len(fields) != 2:
            raise MalformedLineError(f"Bad line format: '{line}'")

        student = fields[0].strip()
        try:
            score = float(fields[1].strip())
        except ValueError as err:
            raise MalformedLineError(f"Non-numeric mark in: '{line}'") from err

        result[student] = score

    return result


with open("marks_ok.txt", "w", encoding="utf-8") as f:
    f.write("Kabir,91\nZara,84\n")

with open("marks_bad.txt", "w", encoding="utf-8") as f:
    f.write("Kabir-91\nZara,notanumber\n")

for filename in ["ghost_file.txt", "marks_bad.txt", "marks_ok.txt"]:
    try:
        parsed = load_marks(filename)
        print(f"{filename} -> {parsed}")
    except (FileNotFoundError, BlankFileError, MalformedLineError) as err:
        print(f"{filename} -> {err}")

Backlash risks

Every skill has a cost when mishandled. These are the ways this one bites back.

Opening with "w" to readWrite mode erases the file instantly. Reading needs "r"; adding needs "a".
Bare except:It hides every error, including your own bugs. Always name the exception you expect.
Forgetting .strip()Every line read from a file ends with \n, so "5\n" == "5" is False.
Catching errors you can't handleIf there is no sensible recovery, let it crash loudly rather than silently continuing with bad data.
